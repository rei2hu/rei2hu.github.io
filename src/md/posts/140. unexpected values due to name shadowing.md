# unexpected values due to name shadowing

Shadowed members can result in unexpected values when using a mix of inheritance
and generics, so if you extend something make sure you really want to create a
new member instead of using the one from the superclass.

I was looking at a piece of code recently that was null for some reason. The
isolated lines looked something like this:

```java
System.out.println(obj.field); // "value";
super.printField();
// public void printField() {
//     System.out.println(obj.field); // null
// }
```

Isn't that weird? The value is `"value"` but turns into `null` after the
function call. This reminds me of `this` binding issues in javascript but this
is java. What could the cause be? Shadowing! A full example of this is as
follows

```java
class Main {
    static class Obj {
        public String field = null;
    }
    static class SubObj extends Obj {
        public String field = null;
        public SubObj(final String f) { super(); field = f; }
    }
    static class GenProcessor<T extends Obj> {
        public final T obj;
        public GenProcessor(final T o) { obj = o; }
        public void printField() { System.out.println(obj.field); }
    }
    static class SpecProcessor extends GenProcessor<SubObj> {
        public SpecProcessor(final SubObj o) { super(o); }
    }

    public static void main(String[] args) {
        final var c = new SubObj("value");
        final var b = new SpecProcessor(c);
        b.printField(); // null
    }
}
```

To debug this, I had attempted to add a `printField()` method to `SpecProcessor`
as I as unable to modify the `GenProcessor` or `Obj` classes.

```java
    static class SpecProcessor extends GenProcessor<SubObj> {
        public SpecProcessor(final SubObj o) { super(o); }
        public void printField() {
            System.out.println(obj.field); // "value"
            super.printField(); // null
        }
    }
```

And this is how I saw what I saw. Pretty weird! In fact, because I couldn't
modify some parts of the code directly, I resorted to attaching a remote
debugger to the application as it ran inside a Docker container. The debugger
showed the following values

```text
this.obj.field = null;
this.obj = SubObj{ field = "value" };
```

That looks weird, right? Though, at the end of the day, once I realized the
issue it makes a lot of sense, since the core idea is that we're looking at
something like this:

```java
class Main {
    static class Obj {
        public String field = null;
    }
    static class SubObj extends Obj {
        public String field = null;
        public SubObj(final String f) { super(); field = f; }
    }

    public static void main(String[] args) {
        final var c = new SubObj("value");
        System.out.println(c.field); // "value"
        System.out.println(((Obj) c).field); // null
    }
}
```

The "fix" was simply to remove `field` from `SubObj`. It was mainly difficult to
figure out because the original code I was working with was much more complex.
Though I think it was pretty interesting to come across something like this.
